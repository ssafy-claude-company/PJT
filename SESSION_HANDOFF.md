## 시스템 한 줄 요약
Discord에서 여러 봇이 "AI 직원"처럼 협업하는 멀티에이전트 시스템. 사용자가 `[Request] To: @봇`
또는 등록 프로젝트 채널에 평문으로 일을 시키면, 담당(리더) 봇이 팀을 꾸려 Task를 만들고 동료에게
나눠 맡겨 산출물을 만들고 배포까지 한다. 핵심 가치 = 탈중앙 협업(리더 독식 금지) + 결과 품질.

## 직전 작업: 타임아웃 관련 결함 3건 — 상태
라이브 검증(P-002 "던전 탈출" 게임 개선)에서 드러난 문제를 교정함.

1. **오래 걸리는 담당자가 중간에 끊기던 문제** → 해결.
   고정 시간이 아니라 "활동이 일정 시간 전혀 없을 때만" 멈춘 것으로 보도록 바꿈. 작업 중이면
   얼마가 걸리든 끊지 않음. (코드+테스트 확인)
2. **끊긴 뒤 정리 관련** → 위 1번으로 발생 조건 자체가 사라짐(작업 중인 담당자를 더는 끊지 않으므로).
   드문 진짜 정지는 `scripts/run_listener.sh`의 자동 재시작이 최후 안전망.
3. **끝났는데 "중단"으로 잘못 닫히던 문제** → 해결.
   진행하던 작업은 보존하고 같은 담당자가 "이어서" 마무리하도록. 정상 마무리 시 상태 자동 복원.
   (코드+테스트 확인)

## 남은 일: 라이브 재검증 (이게 다음 할 일)
수정이 실제로 듣는지 디스코드에서 한 번 더 돌려 확인한다.
- 절차: 사용자가 `#test`(또는 P-002 채널)에 개선 요청을 보냄 → 흐름을 **로그 알림으로만** 관찰.
- 관찰 포인트:
  - 담당자가 오래 걸려도 중간에 안 끊기고 작업을 끝내는가
  - 흐름이 "중단"이 아니라 **정상 완료**로 닫히는가 (`flow_done`의 `comm_done=true`)
  - 진짜 멈춤이 나면 "이어가기"로 작업 보존되며 같은 담당자가 잇는가
- 관찰 방법: `Monitor`로 `logs/flow.jsonl`과 리스너 로그를 `tail`해서 이벤트 알림만 받는다.
  - flow 마일스톤 키워드: `set_goal` · `complete_task` · `deploy` · `flow_done` · `agent_timeout` ·
    `owner_resumable_timeout` · `idle_abort`
  - 리스너 키워드: `메시지 도착` · `요청 수신` · `요청 처리 완료`

## 작업 규칙 (안정적으로 진행하기 위해 — 중요)
이 작업 영역은 시스템 내부를 깊게 다루다 보니, 아래를 지키면 세션이 안정적으로 유지된다.
- **검증은 로그 알림 중심으로.** 실행 상태 확인은 `Monitor`로 로그 파일을 `tail`해서 한다.
- **내부를 줄줄이 파고들거나 길게 분석하지 않는다.** 결론 위주로 짧게.
- **비밀값(토큰·키)을 명령이나 출력에 절대 노출하지 않는다.** 이미 `.env`에 있으니 다시 쓸 일 없음.
- 같은 주제를 반복해서 장황하게 설명하지 않는다.
