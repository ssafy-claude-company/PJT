# Organt 협업 재설계 — 정밀 구조 설계서

> 근거(전수): 캐논 docs(`ssafy-claude-company/docs` — Communication/Task/Project Rule, RFC-002·004·005·006·007·008) + git 350커밋 진화사 + SESSION_HANDOFF(142KB) + 현 코드(`src/*.py`) 전수 감사.
> 목표: **땜질이 아니라** — 단일흐름의 안정성·토큰효율을 유지한 채, 캐논이 설계했으나 드리프트한 "진짜 협업"(P2P 협의·팀 기획·개인 성장·강점 분배)을 *load-bearing*으로 복원하고, 마무리(검증)를 목표지향적으로 수렴시킨다.

## 0. 진단 요약 — 무엇이 어긋났나

캐논 설계 5축 vs 현 라이브(드리프트):

| 축 | 캐논 설계(권위) | 현재(라이브 측정) | 드리프트 원인(커밋) |
|---|---|---|---|
| ① P2P 협의 | "Peer to Peer 소통…중요"(Communication.md). Info는 누구든 직접 | **Info 0건**, 워커 선발화 1건 | a28d62d 교차도메인 게이트 + 워커 기본값 |
| ② 팀이 목표 정의 | "Team이 Purpose로 Goal 정함"(2회), 전원 회의 | 리더 단독 | meet 사용률 ≈0 (권고일 뿐) |
| ③ 리더는 Lead, 구현X | "Work보다 Leading", 대리구현차단(게이트4) | 리더가 다 함 | 1276e1e(도구 복원) 이후 미강제 |
| ④ 개인 기여·성장 | Skill 3채널 + 수면증류 + craft profile | 굶음(워커 "받고 보고"만) | 허브-스포크가 학습 원료 고갈 |
| ⑤ 병렬실행+직렬통합 | GWT: 통합만 직렬, 실행은 fan-out | 병렬 비활성 | 9e8d602(P-029 Write유실) |

**핵심 통찰**: 드리프트는 사고가 아니라 *단일베턴 불변식이 병렬·동료경로를 안전하게 못 받쳐서* 사건마다(P-016/024/029/030/031) 자유도를 빼고 리더로 모은 합리적 후퇴. **협업 도구(meet·vote·peer협의·학습)는 코드에 *아직 다 있다* — 재작성이 아니라 *다시 load-bearing*으로.**

**사용자 철학 정합**: ①~④는 *전부 순차*로 가능(단일흐름 그대로) → 협업 *품질*은 프리미티브 사용에서 나오지 병렬에서 나오지 않음. ⑤(병렬 실행)은 별개 throughput 최적화(P-029로 불안정) → **안정성 게이트 옵션으로 분리, 지금은 보류.** "협동 빈도=품질"이라 ①②를 강제하면 품질이 오른다.

## 1. 설계 원칙 (불변식)

1. **단일흐름·직렬 통합 유지** — 베턴(한 시점 1명), LIFO, 리더=통합/판정 직렬. (캐논 GWT의 "통합은 직렬"과 일치, 사용자 확정)
2. **협업은 순차로** — P2P 협의·기획회의·검증은 베턴이 *옆으로도*(워커↔워커) 흐르는 것이지 동시성이 아님.
3. **강제(load-bearing) > 권고** — RFC-008의 "측정함정"처럼, 권고는 안 쓰임(meet 0). 품질 행위는 *게이트로 강제*하되 *명시 탈출구*를 둠(시늉 방지).
4. **역할 다양성이 품질원** — 같은 모델이라 독립성은 *직군 기준·경험의 차이*에서. → 성장(craft profile)을 먹여야 협업 품질이 오름(선순환).
5. **변경은 단계적·안정성 우선** — 각 단계 테스트+라이브 검증, 회귀 시 즉시 롤백.

## 2. 정밀 변경 설계 (축별 · file/함수 단위)

### 축① + ② — 기획 회의를 load-bearing으로 (팀이 Goal·분해·소유를 *함께* 정의)

**현재**: `set_goal`(guide_tools.py, set_goal 핸들러)은 `participated`(협의 흔적)만 요구 — "빈 핑"은 거르지만 *실질 기획회의*는 강제 안 함. `meet`/`vote`(guide_tools.py, meet/vote 핸들러)는 구현됐으나 사용률 0(권고).

**변경**:
1. **`set_goal` 게이트 강화** — 팀이 2명+이면, set_goal 전에 **`meet`(기획 회의) 1회를 *증거*로 요구**. 회의록(collab_notes)에 각 직군이 "내 도메인 Task·소유"를 발화한 기록이 있어야 통과. 탈출구: `[기획 N/A: 사유]`(단독·자명).
   - *위치*: set_goal 핸들러 진입부에 게이트 추가(percept/acceptance 게이트와 동형). `flow.current.collab_notes`에 직군별 발화 존재 검사.
   - *근거*: Task.md "Team이 Goal 정함", 핸드오프 "협동 빈도=품질".
2. **`meet`를 기획에 자동 유도** — create_task 후 owner 미정 상태에서 리더가 set_goal 시도 시, "먼저 meet로 팀과 분해·소유 합의" 안내(현재의 약한 participated 대신).
3. **분해·소유는 회의의 산물** — meet 회의록에서 각 전문가의 "내 도메인 = 이 Task들, 내가 소유" 선언을 set_goal이 흡수해 Todo/owner 배정의 근거로(리더 독단 금지). 캐논 0ecbbc5 복원.

**리스크/안정**: 게이트는 명시 탈출구가 있어 단독 작업·교착 없음. meet는 1라운드 fork(이미 구현·안전)라 토큰 저렴.

### 축① — P2P 협의 경로를 *유발*(인터페이스 경계에서)

**현재**: Info 위임은 *허용*되나 0건. iface 게이트(guide_tools.py ~2210, `peer_info_pairs` 검사)는 *완료 시점*에 owner↔owner 직접 합의 없으면 보류 — 즉 *사후* 차단이지 *사전* 유발이 아님.

**변경**:
1. **iface 협의를 *사전 유발*** — 워커가 *교차도메인 인터페이스를 쓰는* Work를 받으면(예: 백엔드 API를 프론트가 소비), 위임 본문에 **"맞물리는 owner와 request(Info)로 계약(필드·포맷·타이밍)을 *먼저* 직접 합의하라"**를 주입. 완료 게이트(iface)가 막기 *전에* 협의가 일어나게.
   - *위치*: request 핸들러의 Work 위임 경로 — 대상이 다른 도메인 owner의 산출물과 맞물리면(goal/interfaces에 신호) 협의 안내를 동봉.
2. **워커 프롬프트에 협의 기본값** — organt 인격/턴 프롬프트에 "가정·막힘이 있으면 *혼자 추측 말고* 해당 도메인 동료에게 Info로 확인(권장)" — 기본값을 "고립 실행"에서 "필요시 협의"로.
3. **안전한 동료 Work 경로**(P-030 핑퐁 차단 유지) — 교차도메인 *새 Work*는 리더 조율(현행 pending_coordination + b9c5ea2 SYS 자동조율 유지). 단 *인터페이스 협의(Info)*와 *직전 산출물 보완(redo)*은 동료 직접 허용(이미 그러함). 이로써 "허브-스포크 강제"는 새 Work에만, 협의는 자유.

**리스크/안정**: Info는 베턴 중첩(순차)이라 단일흐름 불변. 새 Work만 리더 경유라 P-030 핑퐁 재발 없음.

### 축③ — 리더 대리구현 차단(게이트4 복원)

**현재**: 리더가 Write/Edit 보유(1276e1e 복원). 흡수 게이트(permissions.py ~186/238, `leader_writes`/`lead_act` 임계)는 *볼륨*으로만 차단 — 임계 아래선 리더가 타 도메인 구현 가능. P-031 리더가 직접 구현함.

**변경**:
1. **소유자 있는 도메인은 리더 구현 금지** — 그 Task에 도메인 owner가 *있으면*, 리더의 그 도메인 파일 Write/Edit를 PreToolUse에서 차단→"owner에게 위임하라"(소유경계 게이트와 동형, 리더 *비*면제). 리더는 *자기 직군*·통합 산출물만 직접(1276e1e 정신 — 같은 직군은 hands-on).
   - *위치*: permissions.py 소유경계 게이트에 "owner 존재 시 리더도 타도메인 차단" 분기. 현재 리더 면제(`actor != flow.leader`)를 *owner가 있을 때만* 해제.
2. **근거**: Task.md "Work보다 Leading", RFC-006 게이트4.

**리스크/안정**: 리더 자기 직군·통합은 허용이라 "리더도 일한다"(REWORK §9) 유지. 명시 탈출구 `[직군초과: 사유]`.

### 축④ — 성장 먹이기 + 강점 기반 분배

**현재**: 학습 루프(role_profiles/role_experience/distill, sys_core + main `_sleep_cycle`)는 작동하나 *원료 부족*(허브-스포크라 워커가 반성·협의를 안 함). craft profile은 *역할* 기준만 키움(개인·강점 아님).

**변경**:
1. **협업이 학습을 먹이게** — meet 회의록·peer 협의·교차검증 *발견*을 `[경험]` 원료로 흡수(핸드오프의 "[경험] 고정 섹션" 확장). ①②로 협의가 늘면 → 학습 원료가 늘어 → craft profile이 풍부해짐(선순환). 캐논 "역할 다른 상호작용이 품질원".
2. **강점 기반 분배(신규)** — set_goal/분배 시, **각 후보 멤버의 craft profile·경험을 리더 프롬프트에 *주입*** → 리더가 "이 Task는 X의 강점(누적 경험)에 맞음"으로 배정. 개인 식별(1봇1직업1기억)이 이미 영속이라, 그 봇의 누적 경험 = 그 *개인의 강점 프로필*.
   - *위치*: sys_core 분배 경로 — owner 후보별 `role_experience`/`role_profiles` 요약을 위임 프롬프트에 동봉.
3. **개인 성장 가시화** — 증류가 *역할*만 아니라 *그 봇 개인*의 경험을 누적(이미 봇별 state). 자기계발(sleep) 시 개인 강점이 깊어짐.

**리스크/안정**: 추가 주입은 토큰 소폭↑이나 분배 품질↑(RFC-005 2-pizza·적임 배정). 증류는 유휴 시(비용 0).

### 축⑤ (마무리 수렴) — 검증을 목표지향적으로: 발견(여럿)/판정(1) + 기억 + 회로차단기

**현재**: 교차검증이 무한 루프(P-031 교차41). 리더가 *여러* QA에 쇼핑, 각 QA는 *새 세션*(기억0), 수렴/해결불가 인지 없음. (내가 추가한 재검증 dedup은 *변경0 재검증*만 막음.)

**변경**(사용자 모델 = 발견 여럿 + 판정 1):
1. **판정자(judge) 1명 고정** — 그 Task의 *검증 판정*은 한 명(구현 owner 아님)이 통합·기억·종결. 여러 QA는 *발견*(다양성), 판정자가 *consolidate*. 리더가 QA를 갈아끼우는 "쇼핑" 차단.
   - *위치*: 검증 위임 경로 — 판정자 id를 Task에 고정(`flow.current.judge`), 재검증은 그 판정자에게만. 영속.
2. **실패 서사 = 판정자의 기억** — 발견된 결함·시도·실패를 collab_notes/judge 컨텍스트에 누적 주입 → 판정자가 "20초 N번·자기웨이크업·sw.js 다 실패 = 해결불가"를 *인지*. (LLM 기억 = 컨텍스트니, 판정자 세션 resume + 서사 주입.)
3. **회로차단기** — 교차검증이 임계(예: 12)를 넘는데 안 닫히면 = "수렴 아니라 루프" → **사용자에게 에스컬레이트**: "N회 검증·미수렴, 해결불가(플랫폼 한계?) 가능 — 수용·마감 / 다른 방향?". 1회만(`loop_escalated` 플래그·영속).
   - *위치*: cross_checks 증가 지점(guide_tools.py ~1234) + 임계 시 `flow.user_channel`로 post(`await self.guide.post`).
4. **품질 루브릭 분리**(RFC-008) — *기능 게이트*(이진: 동작하나)와 *품질 루브릭*(질적: craft profile 기반)을 분리. "측정 가능 목표"가 품질을 떨구는 함정 차단.

**리스크/안정**: 판정자 고정·회로차단기는 토큰을 *줄임*(쇼핑·무한검증 제거). 명시 탈출구(수용/다른방향).

## 3. 단계적 구현 순서 (안정성 우선 · 각 단계 테스트+라이브 검증)

| 단계 | 내용 | 위험 | 검증 |
|---|---|---|---|
| **S1** | 축⑤ 회로차단기 + 판정자 고정 (마무리 수렴) | 낮음(토큰↓) | P-031 재시작 → 즉시 에스컬레이트·닫힘 |
| **S2** | 축③ 리더 대리구현 차단(owner 있을 때) | 중 | 리더가 타도메인 구현 시 차단·위임 |
| **S3** | 축① P2P 협의 유발(iface 사전 + 워커 기본값) | 중 | Info 교환 0→증가, peer_info_pairs 생성 |
| **S4** | 축② 기획회의 load-bearing(set_goal 게이트) | 중 | meet 사용↑, 팀이 Goal 정의 |
| **S5** | 축④ 강점 기반 분배 + 학습 먹이기 | 낮음 | 분배에 craft profile 반영, 경험 누적↑ |
| **S6(보류)** | 축⑤(병렬 실행) — file-lease·기계조인 안정성 *입증 시에만* | 높음 | P-029 Write유실 재발 없음 확인 후 |

각 단계: 단위 테스트 → 라이브 1프로젝트 검증 → 회귀 시 롤백. S1~S5는 *전부 단일흐름 순차* (병렬 불필요).

## 4. 손대지 않는 것 (의도적)

- **단일베턴·LIFO·직렬 통합** — 캐논·사용자 확정 정답.
- **영속화(이번 세션 수정)** — act_by 등 포괄 영속·SIGTERM flush·재위임 차단: 유지.
- **병렬 Work 실행(축⑤)** — P-029 Write유실로 불안정 → 안정성 입증 전 *보류*(throughput 최적화일 뿐, 협업 품질엔 불필요).
- **외부 이전·SNS** — 이 설계 적용·검증 후. (반복사망은 워치독으로 임시 방어 중, 영구는 외부 이전.)

## 5. 한 줄 결론

**캐논이 설계한 진짜 협업(P2P 협의·팀 기획·리더는 Lead·개인 성장·강점 분배)은 *전부 단일흐름 안에서 순차로* 복원 가능하며, 도구는 이미 코드에 있다 — 권고를 *강제(load-bearing)*로 바꾸고, 마무리를 *판정자·기억·회로차단기*로 수렴시키면, "받고 보고하는 고립 실행"이 "협의하고 성장하는 팀"이 된다.** 병렬(속도)은 별개로 분리·보류.
