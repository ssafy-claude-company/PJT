# 작업공간 내구성 · 복구 (workspace durability & recovery)

봇 작업공간(프로젝트 파일)이 **유실되어 봇이 원본에 접근 못 하던 문제**의 재발 방지 기록.

## 왜 유실됐나 (2026-06-29)
- 봇 작업공간은 로컬 파일: 기본 `$ORGANT_PJT/organt_sns_workspace/<pid>` (`ORGANT_PJT` 기본 `/home/user/PJT`).
- 컨테이너 교체·서버 이전 시 이 디렉터리가 통째로 안 넘어오면 사라진다. VPS 이전 때 실제로 옛
  컨테이너의 작업공간이 안 넘어와, 새 러너가 빈 상태로 시작 → 봇이 "원본 접근 차단"이라 보고.

## 왜 이제 안전한가 (3중 내구성)
1. **영속 서버**: 러너는 이제 임시 컨테이너가 아니라 **상시 VPS**에서 돈다 → 재시작·크래시로
   작업공간이 사라지지 않는다(컨테이너 회수 위험 제거).
2. **배포 = 영속 백업**: 배포된 프로젝트의 *전체 소스*는 모노레포의 **`<서비스명>` 브랜치**에
   push된다(Render가 그 브랜치에서 배포 — 항상 최신). 즉 **배포만 하면 그 프로젝트는 git에
   영구 백업**된다. 작업공간이 비어도 거기서 복원 가능.
3. **복구 도구**: 유실돼도 한 번에 되살린다 ↓

## 복구 방법
```bash
# 복원 가능한(배포된) 프로젝트 목록 — 이름 = 배포 브랜치
RENDER_KEY=<키> scripts/recover_workspace.sh --list

# 특정 프로젝트 작업공간 복원
scripts/recover_workspace.sh <서비스명> <대상디렉터리>
#   예) scripts/recover_workspace.sh fps-game-1v1 /home/user/PJT/organt_sns_workspace/fps
```
이전/마이그레이션 후엔 `--list`로 확인하고 프로젝트별로 1회 복원하면 끝.

## 한계 · 권고
- **미배포(진행 중) 작업**은 브랜치 백업이 없으니, 마이그레이션 전 반드시 배포하거나 커밋해 둘 것.
- 경로 안정: 다른 서버로 옮길 땐 `ORGANT_PJT`를 명시적으로 같은 값으로 설정해 작업공간 경로가
  드리프트하지 않게 한다.
