# 수렴 재설계 — Acceptance 게이트 (Rule)

> 협업 흐름이 "끝"으로 수렴하지 못하는 근본 원인과, 하드코딩 백스톱(회로차단기)을
> 구조적 수렴으로 대체하는 설계. 이전/자체 SNS 설계에 반영 대상.

## 1. 문제 (라이브 관측)

흐름이 마무리 단계에서 수렴하지 못하고 **무한 개선 루프**에 빠진다.

- **P-032 (Dungeon Escape)**: 최종 QA 단계에서 리더가 `complete_task` 대신 **VFX 추가(신규 스코프)**를
  계속 위임 → 교차검증 11→13, `deploy_cap` 5회, 회로차단기 파킹. 사용자가 "배포됐냐" 물어도 끝나지 않음.

## 2. 근본 원인

Task에 품질 바가 **둘** 공존하고, 수렴이 **자율**에 맡겨져 있다 (`guide_tools.py` Task 필드):

| 필드 | 의미 | 성질 |
|---|---|---|
| `acceptance` | "좋음(상용)"의 구체·검증가능 기준 (수용 계약) | **유계** — 충족하면 끝나야 할 선 |
| `standard` | exemplar 기준의 *최대* 품질 표준 | **무계** — 항상 더 높은 버전 존재 |

세 가지가 겹쳐 루프가 발생한다:

1. 검증이 무계 `standard`(최대화)에 앵커링 → 주관 품질은 끝이 없어 **늘 "하나 더"**(VFX 추가)가 정당화.
2. `acceptance` 충족이 **강제 마감이 아니라 자율**(`complete_task`는 임의 호출) → 더 얹고 싶으면 안 부름.
3. "끝" 판정이 **봇의 메타인지**에 의존 → 봇은 "해결 불가/한계효용"을 자판정 못 함 → 무한 루프.

**회로차단기**(교차검증 N회 → 1회 에스컬레이트)는 이 공백의 **백스톱(폭주 정지)**일 뿐 수렴을 만들지 않는다.
코드 주석이 자인: *"봇은 해결 불가를 스스로 판정 못 해 무한 검증하므로, 시스템이 메타인지를 대신해 사람에게 넘긴다."*

## 3. 설계 — 구조적 수렴 (세지 말고 구조로 강제)

1. **Acceptance = 검증가능 체크리스트 (`set_goal`에서 확정)**
   - 합의된 *완료정의(Definition of Done)*를 항목화. 각 항목은 객관 검증 가능해야 한다
     (예: "방코드 입장 시 역할 자동배정", "Explorer 좁은 시야", "2인 상태 동기화", "배포 URL 200").
   - 텍스트 선언이 아니라 **게이트가 읽는 구조화 데이터**.

2. **검증은 오직 `acceptance`만 게이트**
   - 교차검증/QA는 acceptance 각 항목의 충족(pass/fail)만 판정.
   - `standard`(최대화)는 **초기 제작 바**로만 사용(첫 구현 품질 끌어올림). **수렴 게이트로는 금지.**

3. **Acceptance 전부 통과 = 마감 강제 + 스코프 동결**
   - 모든 항목 pass → `complete_task` 강제(또는 추가 Work 교차검증 차단).
   - 마무리 단계의 신규 개선(VFX 등)은 **새 Task / 명시 요청으로만** — 슬쩍 못 얹는다.

4. **회로차단기는 본래 역할로 격하**
   - `acceptance`가 1차 수렴 장치. 차단기는 외부 한계(플랫폼 등)·이상치 백스톱(방어 심층화)으로만.

## 4. 배치: Rule 층

수용/수렴 계약은 **매개체 무관 순수 협업 로직** → `communication` / `guide_tools`의 게이트(= Rule).
(앞서 정리한 "질문 vs 지시" 분류가 Guide(디스코드 어댑터) 이슈였던 것과 대비된다.)

## 5. 이전/SNS 반영

- Task 생성 UI에서 **acceptance 체크리스트를 1급 입력**으로 받는다.
- 검증 화면이 항목별 pass/fail을 표시(투명) → "왜 아직 안 끝났나"가 한눈에.
- `complete`(마감)는 전 항목 pass 시에만 활성화 → **구조적으로 멈출 수밖에 없게**.

> 요지: "봇이 알아서 멈추게" 만드는 게 아니라, **acceptance를 구속력 있는 완료정의로 박아 멈출 수밖에 없게** 한다.
> 인간 팀이 *Definition of Done*으로 무한 폴리시를 푸는 것과 같은 원리.

## 6. 사례연구 — P-032 "Dungeon Escape" (부품은 통과, 전체는 비-게임)

**증상:** "2인 협동 게임"인데 **Explorer 혼자 24칸 걸어 2초에 클리어**(Oracle·레버·열쇠·융합 전부 무시).
정적분석(server.js BASE_MAP flood-fill, 레버 0개·문 닫힘) + 라이브 소켓 replay 둘 다로 실증
(`game_clear {time:2}`, Oracle 완전 방치). 즉 **협동이 구조적으로 불필요**.

**왜 안 걸렸나 (검증 고도 오류):** QA 19/19 PASS·`complete_task` 7회가 전부 *부품 단위*
("이동→fog 갱신", "레버→문 반영", "코드에 `socket.on('fusion_event')` 있음")만 검증. acceptance 항목
자체가 부품 체크리스트라, **"솔로로는 못 깬다(협동 강제)"** 같은 *존재이유 테스트*가 단 하나도 없었음.
완료 게이트(`complete_task`의 수용기준 바인딩)는 *잘못된 고도의 기준*을 충실히 검증했을 뿐.

**왜 꼬였나 (구현 과정):**
1. **리더가 맵을 하드코딩으로 넘김** (`00:56` 요청 본문 "맵 데이터 (하드코딩) 17×13 배열 [[0,..]]") —
   "이 맵이 협동으로만 풀리나?"는 누구의 일도 아니었음(설계 아닌 받아쓰기).
2. **index.html 이중 배정 + 창작자=소유자** — 리더가 백엔드(00:56)·프론트(01:10) **둘 다**에게 index.html을
   배정. 백엔드가 서버 스캐폴딩 중 placeholder를 **먼저 생성(01:01)** → `file_owner`가 백엔드로 잠김 →
   프론트/VFX/사운드가 자기 통합 파일을 직접 못 만져 **모든 index.html 편집을 백엔드 경유 Work로 릴레이**
   (백엔드 index.html 편집 12회 = 대부분 "남의 한 줄 붙여주기", 중복 요청 다수).

### 적용된 수정 (이번 커밋, `guide_tools.py`)
- **`set_goal`**: acceptance에 **'존재이유 테스트' 1개 이상**(전체·부정형 — 실패하면 핵심 목적이 깨지는 것)을
  의무화. + **각 산출물·파일은 한 도메인이 소유하도록 계획**(이중 배정 금지, *먼저 만든 자*가 아니라 *도메인
  책임자*가 소유).
- **`complete_task`**: 마감 전 그 존재이유 테스트를 **최종 사용자처럼 end-to-end 실행**해 증거를 남기도록 지시
  (부품 *유무*가 아니라 *전체 목적 달성*을 본다).
- 검증: `tests/test_sys.py` 201개 전부 통과(서술 강화라 로직 무영향).

### 남은 코드-기제 수정 (이전 시 반영)
- **소유권: 창작자-소유 → 선언-소유.** placeholder 생성이 파일을 잠그면 안 된다. 리더의 산출물 배정(또는
  파일↔도메인 선언)이 `file_owner` 시드를 **override**하도록 `sys_core._seed_file_owner` + PreToolUse 훅 개정.
- **존재이유 테스트의 *하드 게이트화*** (선택): `set_goal`에서 acceptance에 부정형/통합 테스트가 안 보이면
  1회 advisory 보류(percept 게이트 패턴) — 지금은 서술 강제까지만.
